How to understand congestion on vSAN

WARNING:Ensure that you fully read and understand this article before taking any action. Also best practice should be to involve VMware ASAP with a S1 service request

IMPORTANT: There is more than one possible explanation for all of the following behaviors. Encountering one or more of the following does not necessarily mean that a log leak has been encountered. If after encountering one of these behaviors you do not see extremely high log utilization, do not assume that you have hit a log leak.

When the vSAN cache SSD encounters a log leak, the issue can manifest itself in a number of ways. One or more of the following behaviors may occur when this condition is encountered. These behaviors will occur when the underlying storage and network devices are healthy and normal. If there are also storage and network issues in the infrastructure, those must be resolved prior to determining if a log leak has been encountered

Maximum congestion is frequently reported in vmkernel logs or the vobd log will report SSD congestion-threshold violations. This behavior may also manifest as congestion messaging, rapidly cycling between normal/violation levels.

 VMs residing on vSAN encounter extremely poor IO performance when physical disk performance is acceptable and all disks are online and available.
• This can also result in Linux VMs residing on vSAN to go read-only.
 Objects that should be available (online mirror(s) and quorum rules are satisfied) fail to open due to I/O error and the ownerAbdicate process does not restore availability.
 Namespace objects that should be available (online mirror(s) and quorum rules are satisfied) fail to appear in the vSAN datastore
 Mounted/opened namespace objects have frequent VMFS heartbeat pulse-set errors or heartbeat timeouts.
• This may also be associated with hosts entering a “not responding” state in vCenter.

vmkernel log heartbeat messaging:

 Resync activity making no progress
 Objects associated with extremely slow IO handling or unexpected loss of availability share a dependency on the same disk group or subset of disk groups.

Why this happens:

When associated with a large buildup of cache SSD log volume, these symptoms are often due to a product flaw in vSAN that can cause log entries in the cache SSD to get stuck due to physical disk failure (early 5.5 releases), inability to process IO associated with a deleted/destroyed object or component, or other reasons. These stuck log entries can ultimately block the logs from draining.

Over time, this log buildup will exhaust the defined logging capacity in the cache-tier SSD. When this occurs, the cache tier for the affected disk group will be unable to process inbound IO operations in a timely manner. This will cause a congestion condition to be raised and latency will be introduced to the inbound IO to allow the cache-tier SSD to catch up. As the log draining process is blocked, however, the logs never reduce and as a result the congestion and high effective latency to the final workload remain present and affect production. This condition will appear even if all underlying hardware (disks, controllers, etc.) are healthy and present as expected.

Once this problem is encountered, there is no patch at this time that will resolve the issue and permit the logs to drain successfully. If a cache-tier SSD log-buildup condition is encountered, there are two available work-arounds described in the "How to resolve" section. If at any time you are unsure of the steps involved in the workaround or you do not fully understand the consequences, engage VMware with a S1 escalation ASAP!

How to resolve:

To improve IO performance when this issue is encountered, the offending disk group needs to be removed from the active IO path or other short-term measures need to be taken. By removing the offending disk group from the IO path, performance will improve and production will be restored.
