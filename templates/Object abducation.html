Owner Abdication Explained

What is owner abdication?

If there are indications that the data should be available (like other components on the same disks are available and working for other objects), we can force DOM to refresh component states by instructing the object owner to surrender its ownership of the object

When the DOM owner releases its ownership claim another node in the VSAN cluster will assume the ownership role for the object. When the object ownership moves to a new node, the component states will be refreshed. If the previously absent components are actually available, the new owner will detect the true state of the underlying components when it initializes its ownership role.

How do we force owner abdication?

There are a couple ways to accomplish owner abdication. Ultimately when we perform an owner abdication we are trying to refresh the state of a component to force it become active. The first and easiest method to accomplish this is from the RVC. You are probably already familiar with the vsan.check_state command that will list all the objects that are currently inaccessible or orphaned on the datastore. If you use the –r (refresh) option with the vsan.check_state command, it will force all VSAN hosts to refresh the state of their objects.

Example:

vsan.check_state . -r

The second way to try and change an object/component’s state is by performing owner abdication against a host. We will need to SSH to a one of VSAN hosts to first identify which host owns the object, and then SSH to the host that is the owner to force it to abdicate.

1. To force the DOM owner to surrender ownership of the component, we must first find the current DOM owner of the object. This step requires you to know the object UUID for the unavailable object. You can get the object UUID from the vsan.check_state command as it will listed the inaccessible objects. If vCenter/RVC is unavailable then you can use the DOM_tool with the –cs option to list the inaccessible objects on the datastore. We can get the object ownership information from cmmds-tool which can be run from any VSAN hosts.
# cmmds-tool find -t DOM_OBJECT -f json -u |head

2. Once you have identified the owner UUID (highlighted in the above example), determine which host is associated with that UUID. We can find the hostname using cmmds-tool.
# cmmds-tool find -t HOSTNAME -f json -u |grep content

3. Once you have identified the host that owns the DOM object, log into that ESXi host via SSH. From the ESXi host, we can run the ownerAbdicate.
This is executed via vsish. # vsish -e set /vmkModules/vsan/dom/ownerAbdicate
