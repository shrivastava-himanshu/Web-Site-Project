vCenter and ESXi communication issues

How to troubleshoot ESXi disconnects when hostd is running

There are a number of things that could be causing an ESXi host to be disconnected in vCenter.
  The first thing to do is to ping the vCenter from the ESXi host to ensure that they can communicate.
The ESXi uses UDP port 902 to maintain a connection with vCenter. Regular heartbeats are sent over this port to maintain a connection.
If a connection cannot be made over port 902 then the ESXi host will disconnect from vCenter.
There are multiple levels that this 902 issue could stem from.
The first one that I would look at is the vpxa.cfg file The vpxa.cfg file is located in /etc/vmware/vpxa.
If you open that file, you should see both the ServerPort and ServerIP entries at the bottom of the file.
These entries represent the communication port between the ESXi and vCenter as well as the vCenter IP address.
To open the file, type:less /etc/vmware/vpxa/vpxa.cfg
At the bottom of the file, you will find these entries.
ServerPort should be set to 902.
ServerIP should be the vCenter managed IP.
  The managed IP should match the IP address returned from running ipconfig on the vCenter Machine Windows command line.

   

After verifying that the ServerIP is set correctly in the vpxa.cfg file, please ensure that port 902 is referenced in vpxa.cfg.
If ServerPort is blank, or it is showing a -1 or 922 value then please change it to 902.
The next thing that we will need to do is verify that the correct vCenter IP is showing on the hosts file.
Location is /etc/hosts.To view it, type less /etc/hosts.
This should have an entry for the vCenter IP, vCenter short name and vCenter Fully Qualified Domain Name.
Ensure that all are correct.

After doing this, if you are connecting the host to vCenter using Hostname or FQDN, please ensure that the host is resolvable by DNS.
To check this, log on to a windows machine in the same domain and open a command prompt.
  Type all of the following combinations to ensure that the ESXi host is fully resolvable:
nslookup HOSTNAME
nslookup FQDN
nslookup IP
If it is not resolving,  then customer will need to resolve the DNS issues internally.

If all of the above steps have been completed and the host is now fully resolvable, please attempt to reconnect the ESXi host to vCenter using the FQDN.
The issue may be resolved due to the changes we have made to /etc/vmware/vpxa/vpxa.cfg, /etc/hosts or the DNS changes. This is depending on how many if not all of the above were misconfigured.

If we still cannot connect, the issue may well be related to port 902 on the physical switch.
They may have a firewall blocking it on the switch.
You can check to see if port 902 is blocked from the Vmware end by logging on to a direct connection to the ESXi host through the vSphere Client.
Click on the ESXi host icon in the left pane > Configuration > Security Profile > Firewall
Look through the Firewall settings and verify whether or not all of the boxes are checked which reference port 902.
If traffic is enabled on UDP 902 (UDP 902 box above is ticked), then it is up to the customer to find what is blocking UDP 902.
To prove that this issue is related to 902, ask the customer to run Wireshark tests between the problematic host and the vCenter.
If it is a VCSA, the customer can test the following command to make sure that 902 is open to the Host:
curl –v telnet://ESXI-IP:902
If the customer has a Windows vCenter, they can install Wireshark on the vCenter machine and perform these tests.Ref this link for more info

From the ESXi host, we can run:
nc –z vcenter-ip 902
After this issue is resolved, the host should connect if UDP 902 is the culprit.

There are also other scenarios which would prevent ESXi hosts from connecting.
Another common scenario would only apply if all hosts are disconnected from vCenter but you can still access them directly through the vSphere Client. This will have happened if vCenter was re-installed after renaming the SSL folders.
We could also see a situation whereby a host has lost its synchronization with vCenter.
As a result, the ESXi hosts will have lost their SSL trust with vCenter and that will need to be re-established.
  To do this, attempt to connect the hosts one by one and when they fail, simply enter the root credentials again to connect. This is needed to update the root password entry in vCenter.
